#### P1-001: FR-015 (timer cancellation) regressed by P2-001 implementation

- Evidence:
  - `src/Iris.Desktop/ViewModels/AvatarViewModel.cs:83-111` ŌĆö `StartSuccessTimer` does `_ = Task.Run(async () => { await Task.Delay(delayMs); ŌĆ” })` with no `CancellationToken`.
  - `src/Iris.Desktop/ViewModels/AvatarViewModel.cs:140-145` ŌĆö `CancelSuccessTimer()` operates on `_successCts`, but `_successCts` is never assigned a new instance anywhere in `StartSuccessTimer`. It remains `null` forever.
  - Spec `docs/specs/2026-04-30-phase-5-5-avatar-v1.spec.md` FR-015: "Timer ąŠčéą╝ąĄąĮčÅąĄčéčüčÅ ą┐čĆąĖ `CancelSuccessTimer()` ą┐čĆąĖ ą║ą░ąČą┤ąŠą╝ ą┐ąĄčĆąĄčģąŠą┤ąĄ čüąŠčüčéąŠčÅąĮąĖčÅ".
  - Original audit `docs/audits/2026-05-01-phase-5-5-avatar-v1.audit.md` line 207-208: "Timer dispose ą┐ą░čéčéąĄčĆąĮ ą║ąŠčĆčĆąĄą║čéąĄąĮ" ŌĆö this no longer holds.
- Impact:
  - `CancelSuccessTimer()` is a no-op. The fire-and-forget Task always runs to completion. Tests T-07 and T-08 pass only because the `if (_state == AvatarState.Success)` guard prevents an unwanted Idle assignment when state has already moved on, not because cancellation works.
  - In production, repeated rapid sends can pile up Tasks in the thread-pool, each waiting full `SuccessDisplayDurationSeconds` then doing a cheap state read. Low-volume so unlikely user-visible, but the documented invariant is broken.
  - Disposal also leaks: `Dispose()` calls `CancelSuccessTimer()` (no-op), and the Task continues to run after disposal, potentially calling `OnPropertyChanged(nameof(State))` on a disposed VM (no `_disposed` check in the Task body).
- Recommended fix:
  - Either (a) assign a new `_successCts = new CancellationTokenSource()` in `StartSuccessTimer`, pass `_successCts.Token` to `Task.Delay`, and respect `OperationCanceledException` in the continuation; or (b) keep the simpler approach and add a `_disposed` check before `_state = AvatarState.Idle` plus a per-call cancellation token to reject stale continuations.
