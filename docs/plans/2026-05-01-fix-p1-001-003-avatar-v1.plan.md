# Implementation Plan: Fix P1-001, P1-002, P1-003 (Avatar v1 Audit Follow-up)

## 1. Plan Goal

Устранить 3 замечания уровня P1 из формального аудита `docs/audits/2026-05-01-fix-p2-001-003-avatar-v1.audit.md`, блокирующие merge:

- **P1-001**: Регрессия инварианта отмены таймера. `_successCts` никогда не назначается, `CancelSuccessTimer()` — no-op. Спецификация FR-015 нарушена.
- **P1-002**: Тест T-04 `StateBecomesSuccessThenIdle` падает с `TaskCanceledException` (5 s polling timeout).
- **P1-003**: T-07 и T-08 проходят по случайным причинам (state guard / большое окно delay), не проверяя реальную отмену.

Также (см. аудит §10) допустимо подобрать P2-005 (мёртвое поле/usings) и P2-006 (silent `catch`) как локальные следствия фикса P1-001 — они вписываются в ту же правку без расширения scope. Включаю их в план как **опциональную часть Phase 1**, чтобы избежать второй итерации над тем же файлом.

Цель: вернуть Avatar v1 в состояние "Ready", сохранив архитектурные границы и поведенческие инварианты спецификации. Минимальные правки в `Iris.Desktop` и `Iris.IntegrationTests`. Никаких изменений в Application/Domain/Shared/Persistence/ModelGateway/Infrastructure.

## 4. Implementation Strategy

Фиксы выполняются **строго последовательно** в порядке из аудита §10:

1. Сначала **Phase 1 (P1-001 production fix)** — это корень регрессии. После него T-04 должен стать детерминированным сам по себе (гипотеза подтверждается тем, что текущий fire-and-forget Task с `await Task.Delay(delayMs)` без токена создаёт race с polling в тесте; при правильно подключенном токене и проверке `_disposed` поведение становится предсказуемым).
2. **Phase 2 (P1-002 verification + опциональный fallback)** — запускаем полный suite. Если T-04 зелёный — фаза завершается без изменений теста. Если красный — детерминируем ожидание через `TaskCompletionSource` на `PropertyChanged`.
3. **Phase 3 (P1-003 strengthening)** — добавляем новый тест (T-16) или усиливаем T-07/T-08 так, чтобы регрессия `_successCts` снова стала видимой.
4. **Phase 4 (Final verification)** — solution build, full test suite, format, forbidden-scope diff.

Стратегия минимизирует число проходов по `AvatarViewModel.cs` (всё в Phase 1) и изолирует тестовые изменения в Phase 2-3.

Phase 0 (Reconnaissance) обязательна: подтвердить актуальное состояние файла, отсутствие локальных правок, поведение `WaitForCallsAsync` и существующий `InternalsVisibleTo`.

### Phase 1 — P1-001: Restore Real Timer Cancellation in AvatarViewModel

#### Goal

Вернуть инвариант FR-015: `CancelSuccessTimer()` действительно отменяет ожидающее переход `Success → Idle`. Параллельно (одна правка, один файл) — устранить P2-005 и P2-006 как побочные обязанности.

#### Files to Inspect

- `src/Iris.Desktop/ViewModels/AvatarViewModel.cs` (после Phase 0).

#### Files Likely to Edit

- `src/Iris.Desktop/ViewModels/AvatarViewModel.cs` (единственный файл этой фазы).

#### Files That Must Not Be Touched

- Любые тестовые файлы (правки тестов идут в Phase 2/3).
- `AvatarOptions.cs`, `AvatarState.cs`, `AvatarSize.cs`, `AvatarPosition.cs`.
- `AvatarPanel.axaml`, `AvatarPanel.axaml.cs`.
- `MainWindow.axaml`, `MainWindowViewModel.cs`.
- `DependencyInjection.cs`, `appsettings.json`.
- Любые проектные файлы (`.csproj`).
- `.agent/*`, `.opencode/*`, `AGENTS.md`.

#### Steps

1. В `StartSuccessTimer()`:
   - Перед `Task.Run` создать локальную копию: `CancelSuccessTimer();` (как сейчас) → затем `_successCts = new CancellationTokenSource(); var token = _successCts.Token;`.
   - Передать `token` в `Task.Delay(delayMs, token)`.
   - Внутри `Task.Run` оборачивать `await Task.Delay` в `try { … } catch (OperationCanceledException) { return; }`.
   - После `await Task.Delay` добавить ранний `if (_disposed || token.IsCancellationRequested) { return; }`.
   - Сохранить guard `if (_state == AvatarState.Success)`.
   - Заменить `catch { … }` (P2-006) на `catch (InvalidOperationException) { … }` (Avalonia "dispatcher not initialized") **или** на `catch (Exception) when (Avalonia.Threading.Dispatcher.UIThread is null)` (если такой вариант компилируется); в крайнем случае оставить общий `catch (Exception)` с комментарием, объясняющим headless-fallback. Никаких пустых `catch` без комментария.
2. В `CancelSuccessTimer()` логика остаётся: `_successCts?.Cancel(); _successCts?.Dispose(); _successCts = null;` (после фикса она наконец заработает).
3. В `Dispose()`:
   - `CancelSuccessTimer()` остаётся.
   - Установить `_disposed = true` **до** `CancelSuccessTimer()` (порядок уже такой, не менять, но проверить, чтобы поздняя continuation видела `_disposed == true`).
4. Удалить неиспользуемый `using System.Linq;` (P2-005). `using System.Threading;` оставить, потому что `CancellationTokenSource` теперь реально используется.
5. Не менять публичный API: `State`, `Size`, `Position`, конструктор, `Dispose()` — без изменений сигнатуры.
6. (Опционально, если потребуется для Phase 3 стратегии (a)): добавить `internal bool IsSuccessTimerActive => _successCts is { IsCancellationRequested: false };` — **только если** Phase 0 подтвердила работающий `InternalsVisibleTo` и Phase 3 решено выполнять через хук. Если нет — не добавлять.

## 9. Risk Register

| Risk | Impact | Mitigation |
|---|---|---|
| `Task.Delay(delayMs, token)` бросает `OperationCanceledException` через `Task.Run` без `try/catch` — может попасть в `TaskScheduler.UnobservedTaskException`. | Medium | Phase 1 шаг 1: явно обернуть в `try/catch (OperationCanceledException) { return; }` внутри `Task.Run`. |
| `Dispatcher.UIThread.Post` всё ещё бросает в headless xUnit, и P1-002 остаётся красным даже после P1-001. | Medium | Phase 2 Branch B: переписать T-04 на `PropertyChanged`-based ожидание. План явно описывает этот fallback-путь. |
| `InternalsVisibleTo` имеет другое имя сборки (`Iris.IntegrationTests` vs `Iris.Integration.Tests`). | Medium | Phase 0 шаг 4 проверяет фактическое имя. Если несовпадение — Phase 3 переключается на стратегию (b). Не редактируем `Iris.Desktop.csproj` (out of scope). |
| Race-condition: `CancelSuccessTimer` вызывается из UI-потока, а continuation `Task.Run` — из thread-pool. Возможно одновременное обращение к `_successCts`. | Medium | `_successCts` остаётся `nullable`, операции `.Cancel()` и `.Dispose()` идемпотентны. После Phase 1 continuation проверяет `token.IsCancellationRequested` рано; гонка между `Cancel()` и `await Task.Delay` уже завершившимся — допустима, потому что `_state != Success` guard остаётся. Если в будущем потребуется строже — добавить `lock`, но **не в этом плане**. |
| Удаление `using System.Linq;` сломает компиляцию из-за где-то скрытого LINQ-вызова. | Low | Phase 1 шаг 4: после удаления — выполнить `dotnet build`. Если красный — вернуть `using`. |
| Сужение `catch` ломает headless-fallback (Phase 1 шаг 1, P2-006). | Low | Если `dotnet test` после сужения снова делает T-04 нестабильным — расширить до `catch (Exception)` с комментарием. Не маскировать молча. |
| T-16 проходит и без P1-001 фикса (false-positive тест). | High (но контролируемо) | Phase 3 Verification содержит mutation-проверку: мысленно откатить Phase 1 и подтвердить, что T-16 падает. Если нет — переформулировать тест. |

## 10. Implementation Handoff Notes

### Critical Constraints

1. **Один файл в production-фазе.** Phase 1 трогает **только** `AvatarViewModel.cs`. Любые "пока я тут, давайте ещё подправим…" — отклонить.
2. **Не возвращаться к `System.Threading.Timer`.** Текущая реализация на `Task.Run`+`Task.Delay` принята; план чинит её, а не отменяет.
3. **Не менять публичный API.** Конструктор, `State`, `Size`, `Position`, `Dispose()` — без изменений сигнатур. `internal` хуки разрешены только если Phase 0 подтвердила `InternalsVisibleTo`.
4. **Не подгонять тесты под код.** Если T-04 после Phase 1 остаётся красным — Phase 2 Branch B обязательно. Не помечать Skip и не ослаблять assertions.
5. **Forbidden scope.** Не трогать `Iris.Application`, `Iris.Domain`, `Iris.Shared`, `Iris.Persistence`, `Iris.ModelGateway`, `Iris.Infrastructure`, `Iris.Api`, `Iris.Worker`.
6. **Memory updates выполняются отдельно** через `/update-memory`, **не** во время `/implement`.

### Risky Areas

- **`OperationCanceledException` propagation** — обязательно ловить в Task.Run continuation, иначе всплывёт в `TaskScheduler.UnobservedTaskException`.
- **`_disposed` race** — `_disposed = true` ставится синхронно в UI-потоке `Dispose()`. Continuation в thread-pool может прочитать его невалидно с т.з. memory model. Для надёжности можно использовать `Volatile.Read`/`Volatile.Write`, **но** это расширение scope; в первой итерации достаточно простой проверки. Записать в `.agent/debt_tech_backlog.md` как opt-in улучшение.
- **`InternalsVisibleTo` name match** — Phase 0 обязана подтвердить точное имя.
